Administrator documentation

Cybersecurity Incident Registry - complete administration guide.

Download administrator PDF

1. Administration scope

Administrators manage users, roles, LDAP, OAuth2/OIDC SSO, MFA, configurable lists, personnel, recommendations, organization data, notification settings, PDF form templates, audit records, export/import and application-wide options.

2. Other settings

Admin → Other settings contains application URL, application time zone, documentation location, privacy-authority non-notification reason, audit-log retention and interface language.

The interface language can be set to Automatic from browser, Italian or English. Automatic mode displays Italian only for Italian browser locale and English for all other locales.

3. Audit

Every operation performed by users, the notification scheduler or automated tasks is recorded with date, time, operation type, actor and details. Admin → Audit is visible only to administrators and provides search across audit fields.

Audit retention is expressed in months, days, hours and minutes, with a default of 6 months. Records older than the configured retention are purged opportunistically after user operations, scheduler runs and full imports.

4. Configurable lists

Action labels include Maximum time (hours) and Exportable by default. The default exportability value is applied automatically when users add new actions to incidents.

5. Notifications

Task-expiry notifications use the application time zone configured in Other settings. Templates may include placeholders for incident name, reference, initial-information time, pending actions, generated time, application name and external URL.

6. Forms and documents

PDF templates are analyzed from their AcroForm fields. Administrators map incident database fields to template fields and configure the font family and size used when generating completed PDFs.

7. Export, import and backup

Full export/import includes application settings, audit records, users, labels, incidents, actions, documents, notification configuration, form mappings and PDF template binaries. Use full export for application-level backups together with database and upload-volume backups.

8. Bilingual maintenance policy

The maintained web interface and the user and administrator documentation must remain available in both Italian and English. Future instructions may be provided in Italian; the corresponding English UI and documentation text must be updated at the same time.

0.110-96 - User notification required

In incident Procedural warnings, the user-notification item is now worded as User notification required. This clarifies that the action must be recorded or generated through the notification workflow when required by the procedure.

9. Package README

The package includes README_en.md as the English counterpart of README.md. Functional, configuration and deployment changes must update both README files together with the online documentation.

Update 0.1.0-98 - Incident PDF reports: times and duration

Incident PDF generation now uses a single date/time formatter, without microseconds and with integer seconds. The PDF summary includes the Duration field, populated through Incident.effective_duration and therefore consistent with the main page.

Update 0.1.0-99 - Incident PDF reports: professional layout

Incident PDF reports have been reformatted with a more professional layout. The beginning of the report shows the application logo; if a logo has been uploaded from the GUI, it is also shown without the "custom logo" wording. If no GUI logo has been uploaded, that slot is omitted. A concise table of contents follows. Section titles are highlighted and kept on the same page as their related information whenever possible, avoiding orphan headings. The footer includes page numbering.

Update 0.1.0-104 - Midnight-based deadline scheduler

The deadline task scheduler uses slots calculated from application-local midnight. Settings → Notifications shows the estimated next send, effective interval, current slot and last automatic execution.

Incident-specific reminders

From the incident detail page, users can schedule reminders with an exact date and time, a custom message, and optional CC addresses. Primary recipients are the personnel associated with the incident.

After a restart, the scheduler recovers all due one-off reminders that have not been sent yet and records automated sends in the audit table.

Update 0.1.0-106 - Paginated audit and procedural controls

Admin → Audit now has configurable pagination: default 20 records, maximum 100. The top of the page shows total records, filtered records and the current chunk. Incidents cannot be closed while active procedural warnings remain. All incident notifications include a direct link to the incident detail page.

Update 0.1.0-108 - Cron notification scheduler

In Admin → Notifications, deadline task notifications now support both regular intervals and specific daily times in HH:MM format. Administrators can use cron mode to define exact sending times while keeping interval slots. The schedule is anchored to midnight in the application timezone, not to the application startup time, and automatic deliveries record slot, outcome and SMTP errors in audit.

Update 0.110-97 - PDF reports and attached documents

Incident PDF report generation uses explicit column widths for the Documents table: the document name is prioritised over the upload date/time column. Timestamps are normalised without microseconds and with seconds always shown as integer values.

Update 0.1.0-100 - Incident PDF reports: logos

Incident PDF report generation no longer labels the GUI-uploaded logo as custom logo. The logo is included only when the configured file exists; otherwise the cell is not generated.

Update 0.1.0-101 - Incident PDF reports: logo image rendering

PDF generation now rasterises SVG logos into temporary PNG files before passing them to ReportLab as images. The initial logo table contains images only, without textual labels; the GUI logo is still conditional on the configured file being present.

Update 0.1.0-102 - Incident PDF reports: application and uploaded logos

Incident PDF generation loads the application logo from the static documentation PNG, which is more reliable for ReportLab, and uses the SVG only as a fallback. The GUI-uploaded logo path is resolved whether absolute or relative; the logo is included next to the application logo only when the file exists.

Notification scheduler and Admin menu

The periodic deadline-task check is executed by an internal scheduler started with the application and does not require active HTTP requests. Every effective run is recorded in the audit log. The Admin menu is organised into collapsible subgroups for easier navigation.

Update 0.1.0-107

The Audit view displays concise and readable operational details. The incident-specific reminders section uses a responsive layout suitable for smartphones.
